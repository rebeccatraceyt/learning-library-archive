Global Variables

Are they really that bad?

You'll have almost certainly heard by now that "global variables are a bad thing." The problem is that a lot of programmers will repeat that exact mantra without explaining why. The truth is that in the context of the projects you'll be building on this course, global variables are probably perfectly fine.

Global vs. Local variables

Before we begin, it is important to distinguish the difference between global and local variables:

Global

Variables that are accessible from any-where in your code

const globalVar = 5

Local

Variables that are only accessible from within the function

function myFunction() {
const localVar = 5
};

A global example

Consider this - if you create a variable at the very top of your JS file called timeZone and set its value to "UTC", that's a global variable:

const timeZone = "UTC"

This makes sense as a globally accessible value, because you can define it once, use it in lots of functions, and if you decide later on that you want to change the time zone, you just have to change one single variable.

In fact, it is inevitable that you'll need to create variables that can be accessed from anywhere in your code. Larger scale settings that will need to be accessed across a range of functions. If you're coding a physics simulation, for example, there are certain things that will be needed throughout the codebase - the force gravity exerts on objects, for example.

This is a fact of coding that's unavoidable. Almost all codebases will have a list of constant values stored somewhere which manage various pieces of behaviour across the application. Your Django projects have a settings.py file, for example.

The problem with global variables arises if you're using external libraries. Suppose you've got your variable called timeZone and you import an external library to help with something like Daylight Savings conversions. The problem is, this library might also have its own variable called timeZone.

And because your timeZone variable is global, it will be accessible everywhere, including in that other library's codebase. Now you've got a problem - the two variables are going to clash and cause unexpected behaviour in your code. And that's why global variables are frowned upon. By their very nature, they're global - not just to your JS files, but any other JS file loaded by your HTML page. And for that reason they can clash with other variables.

But this doesn't change the fact that we mentioned earlier. Pretty much every project will require certain variables to be widely accessible. There are multiple ways to achieve this without making them fully global. You can wrap all of your code in a function, take this Stack Overflow thread for example, which is a somewhat hacky alternative to giving it its own namespace.

How can we work with global variables?

The simplest approach by far is just to recognise why global variables are considered to be "bad," and work around that. In the case of student projects, it likely won't be an issue as you'll be importing a limited number of (if any) external libraries.

For example, you can pick variable names that you know won't clash, or even plan for this by prepending them with some sort of unique identifier:

const _MYPROJECT_GRAVITY = -9.81

So there you go - global variables aren't inherently bad because they're bad coding practice - they're bad because they can pollute your global namespace.
